# DrBones v0.4.0

Release type: project identity, playbooks, agent-governance, and stabilization workflow release.

## Summary

`v0.4.0` is the first release after the repository was renamed to **DrBones**.

This release keeps the PMP / AI Project Skeleton origin doctrine, but makes the public project identity more memorable and practical:

```text
DrBones = public project name
PMP / AI Project Skeleton = legacy/origin doctrine and operating discipline
```

The release also adds vendor-independent playbooks, MCP-style tool-agent governance, outside-agent-pattern distillation, release-readiness stabilization, and cross-repo foreground context loading.

The major theme is:

```text
Do not just add more bones.
Teach the repo how to check whether the skeleton still hangs together.
```

## Highlights

### Repository renamed to DrBones

The public repository is now:

```text
https://github.com/lightrock/drbones
```

The project now uses **DrBones** as the public-facing project identity while preserving PMP language where it describes the original doctrine or compatibility layer.

### Vendor-independent playbooks added

A new `playbooks/` area defines reusable, repo-owned workflow guidance that any capable AI assistant or human can read.

Playbooks are explicitly vendor-independent. They are not Claude Skills, Codex features, ChatGPT memory, or vendor-specific prompt packs.

The repo now distinguishes:

```text
AGENTS.md
  Global operating rules.

workorders/
  One-time task contracts.

examples/day-*/
  Narrative teaching examples.

playbooks/*/PLAYBOOK.md
  Reusable task-specific workflow guidance.
```

### Day 11: playbook packaging

Added:

```text
examples/day-11-playbook-packaging/README.md
```

This example teaches how to convert a proven invocable workflow into a repo-owned playbook without locking the workflow to one AI vendor.

### MCP-style agent design playbook

Added:

```text
playbooks/mcp-style-agent-design/PLAYBOOK.md
```

This playbook teaches how to design tool-using AI agents and MCP-style integrations with explicit authority boundaries, tool contracts, approval gates, proof paths, and completion reporting.

It preserves the core discipline:

```text
AI recommendation is not authority.
A command is not execution.
Execution is not proof.
```

### Day 12: MCP-style tool-agent design

Added:

```text
examples/day-12-mcp-style-tool-agent-design/README.md
examples/day-12-mcp-style-tool-agent-design/real-world-mcp-comparison.md
```

Day 12 explains how to design MCP-style/tool-using agents correctly instead of just wiring random tools into a model.

The companion note compares Day 12 against typical working MCP examples and separates capability demos from authority-governed agent design.

### Day 13: steal the pattern, not the product

Added:

```text
examples/day-13-outside-agent-pattern-distillation/README.md
```

Day 13 teaches how to learn from outside agent examples without copying unsafe products, hype, vendor assumptions, or bad authority boundaries.

The governing phrase is:

```text
Steal the pattern, not the product.
```

It includes a softened trend-to-product example, ops triage example, browser-testing example, and AI-search visibility example.

### Day 14: stop adding bones and check the skeleton

Added:

```text
examples/day-14-release-readiness-stabilization/README.md
```

Day 14 teaches a release-readiness stabilization workflow for checking whether README files, AGENTS.md, examples, playbooks, wiki pages, checks, workorders, release notes, and authority boundaries still agree after rapid repo growth.

The lifecycle is:

```text
invent
→ capture
→ link
→ check
→ stabilize
→ release
```

### Day 15: cross-repo foreground context loading

Added:

```text
examples/day-15-reference-repository-context/README.md
```

Day 15 teaches how to make a foreground AI smarter by deliberately loading selected related repositories as source context while preserving current-repo authority.

The governing phrase is:

```text
Load more context.
Keep sharper boundaries.
```

It uses the PFEM-family repositories and SkyWrong as example source repositories, but the workflow is generic: classify each related repo's role, extract transferable patterns, mark non-transferable/domain-specific material, and decide whether anything belongs in the current repo.

### Combined rename and release-readiness workorder

Added:

```text
workorders/2026-05-26-0405-by-lightrock-verify-drbones-rename-release-readiness.md
```

## Known limitations

This release was prepared through GitHub file operations, not from a local checked-out runtime environment.

Before tagging or publishing a GitHub Release object, run the standard local checks from the repository root:

```text
python tools/pmp_check.py --area all
python -m pytest
```

If those checks reveal issues, fix them before publishing the GitHub release object.

The GitHub connector used for this release note does not expose a native `create release` action. This file records the release content in the repo; the GitHub Release object and tag should be created in GitHub after local checks pass.
