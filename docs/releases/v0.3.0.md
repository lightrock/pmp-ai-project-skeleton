# PMP AI Project Skeleton v0.3.0

Release type: governance hardening release.

## Summary

`v0.3.0` turns PMP from a mostly doctrine-only skeleton into a skeleton with its first enforceable governance layer.

This release keeps the core idea intact: the repository is the durable project memory, the workorder is the task contract, and AI tools are interchangeable executors or assistants.

## Highlights

### AI coding tools are interchangeable executors

The README now clarifies that PMP is not another AI coding tool. PMP is the repo-native operating discipline that makes AI coding tools interchangeable.

The durable memory belongs in the repository, not in one chat window, proprietary notebook, prompt wrapper, or vendor session.

### Workorder governance is now explicit

Workorder guidance now says required checks are not decorative. A coding agent or executor should keep working until required checks pass unless it is blocked by missing authority, missing access, unsafe ambiguity, or a real conflict with the workorder or repo doctrine.

Completion notes must report what changed, what checks ran, what passed or failed, what was not run and why, whether lessons learned were created or not needed, open questions, and the exact workorder path.

### Lessons learned are part of the default loop

Foreground agents are now expected to include lesson-learned governance when creating workorders.

The coding agent or executor should record lessons learned when work reveals a repeated mistake, missing rule, fragile workflow, ambiguous command, misleading document, questionable assumption, or architecture boundary that the repo should remember.

### Foreground agents must check themselves

Foreground agents now have an explicit self-check rule: if a task is turning into many file edits, repeated repo mutations, test runs, debugging, or environment verification, the foreground agent should stop and create a workorder for an executor instead of trying to perform an implementation chain from chat or a read-only connector.

### Foreground output discipline added

Foreground agents should not accidentally switch into image, slide, document, or artifact generation unless the human explicitly asks for that output type.

They should not offer multiple competing chat outputs and ask which one the human likes best. They should produce one coherent recommendation, one copy/paste artifact, or one workorder.

Copy/paste prompts for coding agents should be clean, self-contained, and free of assistant-side comments, citations, markdown noise, or placeholders that could break execution or change meaning.

### First governance checker added

This release adds a small standard-library Python checker:

```text
python tools/pmp_check.py --area all
```

The checker validates the first workorder governance contract:

```text
schemas/workorder-contract.json
```

It checks workorder filename rules, forbidden rolling names, required governance docs, and completion-note governance discoverability.

### Tests added

The first tests for the governance checker live at:

```text
tests/test_pmp_check.py
```

Run them with:

```text
python -m pytest
```

## Known limitation

This release was prepared through GitHub file operations, not from a local checked-out runtime environment.

The repository now includes the checker and tests, but the final local commands still need to be run by Codex or a local shell:

```text
python tools/pmp_check.py --area all
python -m pytest
```

If those checks reveal issues, the executor should fix them within scope or report the precise blocker.
